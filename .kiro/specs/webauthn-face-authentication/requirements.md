# Requirements Document

## Introduction

This document specifies requirements for implementing WebAuthn-based biometric authentication (Face ID, Windows Hello, etc.) in the chat application. The feature enables passwordless authentication using device biometrics while maintaining backward compatibility with existing username/password authentication. It includes special provisions for admin account setup and prevents credential reuse across multiple accounts.

## Glossary

- **WebAuthn_System**: The backend and frontend components that implement the Web Authentication API for biometric authentication
- **Authenticator**: A hardware or software component (e.g., Face ID, Windows Hello, fingerprint reader) that performs biometric verification
- **Credential**: A cryptographic key pair stored on the user's device and associated with their account
- **Challenge**: A random value generated by the server to prevent replay attacks during authentication
- **Relying_Party**: The chat application server that relies on the Authenticator to verify user identity
- **Admin_Setup_Token**: An environment variable secret used to gate access to the admin account creation endpoint
- **Session_Store**: An in-memory data structure with mutex protection for storing authentication challenges
- **User_Handle**: A unique identifier for the user stored in the WebAuthn credential

## Requirements

### Requirement 1: WebAuthn Registration

**User Story:** As a new user, I want to register using my device's biometric authentication, so that I can create an account without remembering a password

#### Acceptance Criteria

1. WHEN a user initiates registration, THE WebAuthn_System SHALL generate a unique challenge and store it in the Session_Store
2. THE WebAuthn_System SHALL return registration options containing the challenge, Relying_Party information, and user details to the frontend
3. WHEN the frontend receives registration options, THE WebAuthn_System SHALL invoke the browser's WebAuthn API to prompt for biometric authentication
4. WHEN the Authenticator completes verification, THE WebAuthn_System SHALL receive a credential containing a public key and credential ID
5. THE WebAuthn_System SHALL verify the credential signature matches the challenge
6. THE WebAuthn_System SHALL store the credential ID, public key, and user association in the webauthn_credentials table
7. WHEN credential storage succeeds, THE WebAuthn_System SHALL create the user account with the selected display name
8. THE WebAuthn_System SHALL add the new user to the Main_Group_Chat automatically
9. THE WebAuthn_System SHALL return a JWT token for immediate session establishment

### Requirement 2: WebAuthn Login

**User Story:** As a registered user, I want to log in using my device's biometric authentication, so that I can access my account quickly and securely

#### Acceptance Criteria

1. WHEN a user initiates login, THE WebAuthn_System SHALL generate a unique challenge and store it in the Session_Store
2. THE WebAuthn_System SHALL retrieve all credential IDs associated with existing users
3. THE WebAuthn_System SHALL return login options containing the challenge and allowed credential IDs to the frontend
4. WHEN the frontend receives login options, THE WebAuthn_System SHALL invoke the browser's WebAuthn API to prompt for biometric authentication
5. WHEN the Authenticator completes verification, THE WebAuthn_System SHALL receive an assertion containing the credential ID and signature
6. THE WebAuthn_System SHALL verify the assertion signature matches the challenge using the stored public key
7. WHEN verification succeeds, THE WebAuthn_System SHALL identify the user by the credential ID
8. THE WebAuthn_System SHALL update the user's last_seen timestamp
9. THE WebAuthn_System SHALL return a JWT token containing user_id, username, display_name, and role

### Requirement 3: Credential Uniqueness Enforcement

**User Story:** As a system administrator, I want to prevent users from registering multiple accounts with the same biometric credential, so that account abuse is minimized

#### Acceptance Criteria

1. WHEN a user attempts registration, THE WebAuthn_System SHALL check if the credential ID already exists in the webauthn_credentials table
2. IF the credential ID exists, THEN THE WebAuthn_System SHALL return an error message indicating the credential is already registered
3. THE WebAuthn_System SHALL enforce a unique constraint on the credential_id column in the database
4. WHEN a duplicate credential is detected, THE WebAuthn_System SHALL prevent account creation and return HTTP status 400

### Requirement 4: Admin Account Setup

**User Story:** As a system administrator, I want a secure one-time setup flow for creating an admin account, so that I can manage the chat application

#### Acceptance Criteria

1. WHERE the admin setup endpoint is accessed, THE WebAuthn_System SHALL require an X-Admin-Setup-Token header
2. THE WebAuthn_System SHALL compare the provided token against the ADMIN_SETUP_TOKEN environment variable
3. IF the token is missing or incorrect, THEN THE WebAuthn_System SHALL return HTTP status 403
4. WHEN the token is valid, THE WebAuthn_System SHALL allow registration with role set to "admin"
5. THE WebAuthn_System SHALL follow the same WebAuthn registration flow as regular users
6. THE WebAuthn_System SHALL store the admin credential in the webauthn_credentials table
7. THE WebAuthn_System SHALL not add the admin user to the Main_Group_Chat automatically

### Requirement 5: Challenge Management

**User Story:** As a security-conscious developer, I want authentication challenges to be securely managed, so that replay attacks are prevented

#### Acceptance Criteria

1. WHEN a challenge is generated, THE WebAuthn_System SHALL create a cryptographically random 32-byte value
2. THE WebAuthn_System SHALL store the challenge in the Session_Store with the username as the key
3. THE Session_Store SHALL use mutex locks to prevent concurrent access issues
4. WHEN a credential verification request is received, THE WebAuthn_System SHALL retrieve the challenge from the Session_Store
5. THE WebAuthn_System SHALL delete the challenge from the Session_Store after verification succeeds or fails
6. IF no challenge exists for the username, THEN THE WebAuthn_System SHALL return an error indicating invalid or expired challenge

### Requirement 6: HTTPS Requirement

**User Story:** As a security-conscious developer, I want WebAuthn to only function over HTTPS, so that credentials cannot be intercepted

#### Acceptance Criteria

1. WHEN the application runs in production, THE WebAuthn_System SHALL require HTTPS for all WebAuthn endpoints
2. THE WebAuthn_System SHALL configure the Relying_Party origin to match the HTTPS domain
3. IF a WebAuthn request is made over HTTP in production, THEN THE WebAuthn_System SHALL return an error
4. WHERE the application runs in development mode, THE WebAuthn_System SHALL allow localhost HTTP connections for testing

### Requirement 7: Backward Compatibility

**User Story:** As an existing user, I want to continue using username/password authentication, so that I am not forced to use biometric authentication

#### Acceptance Criteria

1. THE WebAuthn_System SHALL maintain all existing password-based authentication endpoints
2. WHEN a user has both password and WebAuthn credentials, THE WebAuthn_System SHALL allow login using either method
3. THE WebAuthn_System SHALL not modify the existing users table schema for password_hash
4. THE WebAuthn_System SHALL allow users to register with password-only authentication if they choose
5. WHEN a user logs in with password, THE WebAuthn_System SHALL return the same JWT token format as WebAuthn login

### Requirement 8: Frontend Integration

**User Story:** As a user, I want a seamless biometric authentication experience in the browser, so that I can authenticate quickly

#### Acceptance Criteria

1. WHEN the login page loads, THE WebAuthn_System SHALL display a "Sign in with Face ID" button
2. WHEN the register page loads, THE WebAuthn_System SHALL display a "Register with Face ID" button
3. WHEN a user clicks the biometric button, THE WebAuthn_System SHALL call the backend to retrieve authentication options
4. THE WebAuthn_System SHALL invoke navigator.credentials.create() for registration
5. THE WebAuthn_System SHALL invoke navigator.credentials.get() for login
6. WHEN the browser prompts for biometric authentication, THE WebAuthn_System SHALL display appropriate loading states
7. IF the browser does not support WebAuthn, THEN THE WebAuthn_System SHALL hide biometric buttons and show only password options
8. WHEN authentication succeeds, THE WebAuthn_System SHALL store the JWT token in localStorage and redirect to the chat page

### Requirement 9: Error Handling

**User Story:** As a user, I want clear error messages when biometric authentication fails, so that I understand what went wrong

#### Acceptance Criteria

1. IF the user cancels biometric authentication, THEN THE WebAuthn_System SHALL display "Authentication cancelled"
2. IF the credential verification fails, THEN THE WebAuthn_System SHALL display "Authentication failed. Please try again"
3. IF the challenge has expired, THEN THE WebAuthn_System SHALL display "Session expired. Please try again"
4. IF the credential is already registered, THEN THE WebAuthn_System SHALL display "This device is already registered to another account"
5. IF the browser does not support WebAuthn, THEN THE WebAuthn_System SHALL display "Your browser does not support biometric authentication"
6. WHEN a network error occurs, THE WebAuthn_System SHALL display "Network error. Please check your connection"

### Requirement 10: Database Schema

**User Story:** As a developer, I want a proper database schema for storing WebAuthn credentials, so that credential data is persisted correctly

#### Acceptance Criteria

1. THE WebAuthn_System SHALL create a webauthn_credentials table with columns: id, user_id, credential_id, credential_data, created_at
2. THE WebAuthn_System SHALL enforce a foreign key constraint from user_id to users(user_id) with ON DELETE CASCADE
3. THE WebAuthn_System SHALL enforce a unique constraint on credential_id
4. THE WebAuthn_System SHALL store the credential public key and metadata as JSON in the credential_data column
5. WHEN a user is deleted, THE WebAuthn_System SHALL automatically delete associated credentials via cascade

### Requirement 11: Session Token Generation

**User Story:** As a developer, I want WebAuthn authentication to generate the same JWT tokens as password authentication, so that session management is consistent

#### Acceptance Criteria

1. WHEN WebAuthn authentication succeeds, THE WebAuthn_System SHALL generate a JWT token using the existing createJWT function
2. THE WebAuthn_System SHALL include user_id, username, display_name, and role in the JWT claims
3. THE WebAuthn_System SHALL set the token expiration to 60 minutes
4. THE WebAuthn_System SHALL sign the token with the JWT_SECRET_KEY from environment variables
5. THE WebAuthn_System SHALL return the token in the same JSON format as password login: {access_token, token_type, user}

### Requirement 12: Display Name Selection

**User Story:** As a new user registering with biometrics, I want to select my display name from available options, so that I have a unique identity in the chat

#### Acceptance Criteria

1. WHEN a user initiates WebAuthn registration, THE WebAuthn_System SHALL require a display_name parameter
2. THE WebAuthn_System SHALL validate the display_name against the predefined list of 50 names
3. THE WebAuthn_System SHALL check that the display_name is not already taken
4. IF the display_name is invalid or taken, THEN THE WebAuthn_System SHALL return an error before initiating WebAuthn ceremony
5. THE WebAuthn_System SHALL store the display_name in the users table with a unique constraint

### Requirement 13: Admin Setup Frontend

**User Story:** As a system administrator, I want a dedicated admin setup page, so that I can create the admin account securely

#### Acceptance Criteria

1. THE WebAuthn_System SHALL provide a frontend page at /admin-setup
2. WHEN the admin setup page loads, THE WebAuthn_System SHALL prompt for the admin setup token
3. THE WebAuthn_System SHALL include the token in the X-Admin-Setup-Token header for all admin registration requests
4. WHEN admin registration succeeds, THE WebAuthn_System SHALL redirect to the login page
5. THE WebAuthn_System SHALL display appropriate error messages if the token is invalid

### Requirement 14: Credential Data Serialization

**User Story:** As a developer, I want WebAuthn credential data to be properly serialized, so that it can be stored and retrieved from the database

#### Acceptance Criteria

1. THE WebAuthn_System SHALL serialize the credential public key, attestation type, and AAGUID to JSON format
2. THE WebAuthn_System SHALL store the serialized JSON in the credential_data column as TEXT
3. WHEN retrieving credentials for authentication, THE WebAuthn_System SHALL deserialize the JSON back to the credential object
4. THE WebAuthn_System SHALL handle serialization errors gracefully and return HTTP status 500 with an error message

### Requirement 15: WebAuthn Library Configuration

**User Story:** As a developer, I want the WebAuthn library properly configured, so that authentication works correctly across different environments

#### Acceptance Criteria

1. THE WebAuthn_System SHALL initialize the go-webauthn library with the Relying_Party ID matching the domain
2. THE WebAuthn_System SHALL set the Relying_Party display name to "Chat Application"
3. WHERE the application runs on localhost, THE WebAuthn_System SHALL set the origin to "http://localhost:3000"
4. WHERE the application runs in production, THE WebAuthn_System SHALL read the origin from the WEBAUTHN_ORIGIN environment variable
5. THE WebAuthn_System SHALL configure user verification preference to "preferred"
6. THE WebAuthn_System SHALL configure attestation preference to "none" for privacy
